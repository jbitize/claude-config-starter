# User-level CLAUDE.md

이 파일은 모든 Claude Code 세션·프로젝트에 자동 로드되는 사용자 전체 instruction이다.
프로젝트별 instruction은 각 프로젝트의 `CLAUDE.md`에 둔다.

이 CLAUDE.md 는 여러 프로젝트에서 시행착오 끝에 정착된 룰의 일반화 버전이다.
본인 취향대로 자유롭게 수정하라 — 톤, skill 트리거 단어, 워크플로우 룰 모두 본인 환경에 맞게 바꿔도 된다. 이건 시작점이지 정답이 아니다.

---

## 🚫 사용자가 할 수 있는 일이라도 내가 할 수 있으면 내가 한다 (최우선 룰)

사용자에게 명령 한 줄 던지면서 "직접 실행하시면 됩니다"라고 떠넘기지 않는다. 내가 도구로
실행 가능한 작업이면 — Bash, Edit, Write, curl, git, python, pytest, gcloud,
xcodebuild 등 — 내가 직접 실행하고 결과를 보고한다. 사용자는 매번 명령 copy-paste 하는
부담을 지지 말아야 한다.

**판단 기준 — 내 영역 (사용자에게 시키지 말 것)**:
- CLI 명령 실행 (python script, git, curl, pytest, npm, bash script 전부)
- 파일 편집 / 생성 / 삭제
- HTTP 요청 (curl, WebFetch)
- 빌드 / 테스트 실행
- 외부 API 호출 (GitHub gh, Slack webhook 등)
- DB 쿼리 (DATABASE_URL 통한 psql, 또는 admin endpoint)
- git rebase, merge, branch 관리

**사용자만 가능한 영역 (시켜도 OK)**:
- 인증 입력 (비밀번호, 2FA, OAuth 로그인 처음 한 번)
- 물리 디바이스 액션 (iOS 앱 강제 종료, 하드웨어 버튼, 디바이스 연결)
- 사람의 의사결정·취향·실측 판단 (UX 선호, 자료 확인, 운영 결정)
- 외부 서비스 GUI 액션 (App Store Connect, Stripe dashboard 등 GUI 전용)
- 사용자 본인의 인지 / 검증 (작동 여부 시각 확인, 진동/소리 확인)

**Anti-pattern 예시**:
- "사용자가 직접 실행하시는 게 정석입니다 — `cd ... && python ...`" — Bash로 내가 실행 가능
- "사용자가 curl로 확인하시면..." — WebFetch나 Bash curl로 내가 가능
- "이 명령 한 줄 돌려주세요" — 돌려주세요 패턴 = 거의 모든 경우 anti-pattern

**올바른 패턴**:
- 내가 Bash로 명령 실행 후 결과 보고
- 결과에 따라 다음 단계 자동 진행 (또는 사용자 결정 영역만 묻기)
- 사용자가 직접 실행하셔야 하는 진짜 경우(인증, 물리 액션)만 명확하게 안내

**Why**: 사용자 부담 줄이기 + 작업 사이클 일관성. 사용자가 매번 명령 copy-paste 하면 작업
효율 떨어지고 sync 누락 가능. 실제 프로젝트 시행착오에서 이 anti-pattern 이 반복돼서
절대 룰로 격상됨.

## 🚨 작업 trigger — Skill 강제 호출 (최우선 룰)

사용자가 다음 의도를 표현하면 **즉시 해당 Skill을 호출**한다. 답변 작성 전에 Skill
호출이 선행되어야 한다. 예외 없음. 짧은 명령("디버깅해", "구현해" 한 마디라도) 도
동일하게 적용. trigger word 일부만 매칭돼도 호출.

| 사용자 표현 | 호출할 Skill |
|---|---|
| "plan 작성", "기획해", "계획해", "설계해", "스펙 만들어", "design", "draft a plan" — 또는 큰 새 기능/변경 요청을 받았을 때 (간단한 수정은 plan 없이 직접) | `plan-create` |
| "plan 검증", "plan 확인", "plan 검토", "plan 체크해", "베스트 plan 인지", "빠진 거 없어", "오버한 거 없어", "테스트 충분해", "review plan", "is this best", "anything missing" | `plan-verify` |
| "구현해", "구현 진행", "구현 시작", "구현해줘", "implement", "make it", "build it", "code it", "코딩해", "코드 작성해" — 사용자 명시 승인 후에만 | `implement` |
| "디버깅", "디버그해", "원인 파악", "원인 찾아", "왜 안 돼", "왜 안 나와", "에러 나", "에러야", "버그 찾아", "버그야", "이상해", "잘못됐어", "fail", "broken", "not working", "why is this", "what's wrong" — 또는 에러 스크린샷/로그 공유 | `find-cause` (옛 이름 `debug` 는 Claude Code reserved name 과 충돌해 rename 됨) |
| "commit해", "push해", "올려", "올려줘", "deploy해", "배포해", "release", "ship it", "코드 올려" — 또는 implement skill 마무리에서 배포 단계 진입 시 | `commit-deploy-flow` |

**위반 anti-pattern** — trigger 표현 받았는데 Skill 호출 없이 바로 답한 경우.
이런 답변은 잘못된 답변이고, 사용자가 지적하면 즉시 재작성한다.

각 Skill의 룰을 매번 그대로 따른다. Skill 호출 후엔 그 Skill의 본문(SKILL.md)이
context에 로드되니, 룰 누락 없이 작업 가능.

**Skill 위치:** `~/.claude/skills/<skill-name>/SKILL.md`. 위 5개 Skill 외에 다른
trigger도 발견되면 새 Skill 추가를 사용자에게 제안.

## 💬 사용자 응답은 알기 쉽게 풀어서 설명

사용자에게 답변할 때는 다음 톤·구조를 따른다.

**핵심 원칙** — 친구나 동료에게 말로 설명하듯 자연스럽게. 표·매트릭스·번호 매김
같은 빽빽한 구조는 정말 비교가 필요할 때만. 평소 답변은 풀어 쓴 완성 문장으로
부드럽게 흘러가도록.

**구체적인 가이드:**

- bullet list 가 정말 필요한 경우 (3개 이상 병렬 항목) 만 사용. 그 외에는 단락
  형태로 풀어 쓴다.
- 표는 정말 비교 축이 명확할 때만. 그 외엔 단락 안에 자연스럽게 녹여 쓴다.
- 굵은 글씨도 한 단락에 2-3개 이하로 절제. 핵심 결론 또는 라벨에만.
- 영문 용어를 쓸 때 그냥 본문에 자연스럽게 섞어 쓴다 (italic 없이).
- 결론은 응답 마지막에 한 두 문장으로 명확히 던지되, 거기까지 풀어 쓴 설명이
  자연스럽게 도착하도록.

**리포트·문서 작성과의 구분:** 리포트나 공식 문서는 표·bullet 으로 정리된
구조가 적합한 곳도 있음 (아래 markdown 포매팅 룰 8개 참고). 하지만 사용자에게
직접 대화할 때의 응답은 그 룰과 별개로 더 풀어 쓰는 톤을 따른다.

**How to apply:** 응답을 작성하기 전에 "이걸 친구에게 카페에서 말로 설명한다면
어떻게 할까?" 를 떠올린다. 표를 그리고 싶어지면 그 비교 내용을 풀어 쓴 두세
문장으로 대체 가능한지 먼저 고민한다.

## 📐 리포트·문서 작성 룰 — Markdown 8개 규칙 (필수 준수)

리포트·분석·기술 문서 (md 파일, plan.md, research.md 등 사용자가 읽을 산출물)
를 작성할 때 **구조는 bullet/표/헤더로 잡고, 각 bullet 안 내용은 반드시 풀어 쓴
완성 문장으로 채운다**. 짧은 라벨 또는 키워드 나열만 있는 bullet 은 금지. 동시에
모든 내용을 단락으로만 풀어쓰는 것도 금지 — 그러면 구조가 사라진다. 둘의 균형
— 구조는 bullet/표/헤더로, 내용은 풀어 쓴 완성 문장으로 — 가 한 번 읽고 이해되는
동시에 다시 보기 좋은 리포트의 핵심.

**How to apply:**

- **쓰기 전** — 헤더/bullet 구조를 먼저 잡고, 각 bullet 안에 1문장 이상의 풀어
  쓴 본문을 채울 수 있는지 확인. 한 줄 라벨만 떠오르면 그 항목은 풀어쓰기에
  필요한 내용 깊이가 부족하다는 신호 — 더 조사하거나 다른 bullet 과 합친다.
- **쓴 후** — 모든 bullet 을 훑어서 1문장 미만으로 끝나는 게 있으면 풀어 쓰거나
  부모 단락으로 흡수. 동시에 단락 5개 이상이 연속되면 그중 일부는 bullet 으로
  구조화할 수 있는지 점검.

위의 풀어쓰기 톤은 유지하되 시각적 단위가 깨지지 않도록 아래 8개 표준을 반드시
따른다.

### 1. 단일성 (atomic bullet)

bullet 한 항목에는 하나의 사실 또는 하나의 주장만 담는다. em-dash (`—`) 로
여러 문장을 사슬처럼 잇지 않는다 (한 항목 안 최대 1회).

- **나쁜 예:** `- 모델이 60%에서 100%로 올랐습니다 — 벤치마크 수명이 짧아졌습니다 — 동시에 Arena에서 4사가 25 Elo 안에 군집했습니다.`
- **좋은 예:** 두 개의 별도 bullet 으로 분리. 각 bullet 은 한 문장 또는 짧은 두 문장.

### 2. 라벨/본문 분리

라벨 + 해설 구조는 굵은 라벨을 한 줄로 끝내고, 해설은 다음 줄 (단락 또는
sub-bullet) 로 내린다. `**라벨.** 본문…` 같이 같은 줄에 모두 붙이는 패턴은
짧은 한 문장에만 허용한다.

```markdown
- **1.1 World-class faculty (세계 수준의 교수진) 에 대한 투자**

  핵심 메시지는 교수진의 시간·자원·영감을 늘려서 창의와 혁신이 꽃피게 한다는
  것이다.
```

**Before / After 예시 — 짧은 라벨 vs 풀어쓴 본문:**

나쁜 예 — 짧은 라벨 bullet 만 나열 (의미가 끊김):

```
- a. EV Chargers. 전기차 충전기 설치 안건.
- b. Metering. 전기 사용량 측정 방식.
```

좋은 예 — 각 bullet 안에 풀어 쓴 본문 (한 번 읽고 이해됨):

```
- **a. Installing additional chargers.** 추가 charger 의 설치 여부와 절차다.
  기존 규정을 검토하면 설치 자체는 허용되지만 위치나 전기 부담 방식이 계약서에
  묶여 있는지 확인이 필요하다.
- **b. Metering and billing considerations.** 전기 사용량 측정과 요금 부과
  방식이다. 전체 예산에서 전기 항목이 크게 인상되었는데 EV 사용자가 늘면 이
  항목이 더 빠르게 증가하므로, 개별 metering 으로 가는지 확인할 필요가 있다.
```

또 다른 나쁜 예 — 구조 없이 단락으로만 풀어씀 (스캔이 어려움) — 정보가 다
들어있어도 시각적으로 어디가 sub-item 의 경계인지 잡히지 않아 다시 읽을 때
스캔이 어렵다. 구조 (bullet) + 내용 (풀어쓴 문장) 의 균형이 핵심.

### 3. List vs. 단락

병렬 항목이 3개 이상이면 list, 1-2개이거나 서사 흐름이면 단락. 단락 안에
"첫째·둘째·셋째" 를 평문으로 늘어놓을 때 4개 이상이 되면 반드시 list 로
변환한다.

### 4. Numbered vs. Bulleted

numbered (`1. 2. 3.`) 는 순서가 의미 있을 때와 외부 참조 번호가 필요할 때만
사용한다. 그 외 모든 병렬 항목은 bulleted (`-`).

### 5. 표 (table)

표는 비교가 핵심일 때만 사용. 셀 안에 한 문장 이상 들어가야 하면 표 대신 list
로 바꾼다. 셀은 약 30-40자 (한국어) 안에 끝나도록 압축하고, 긴 해설은 표 아래
단락으로 푼다.

### 6. 들여쓰기·줄바꿈

- 단락 (paragraph) 사이는 빈 줄 1줄.
- list 항목 사이에는 빈 줄을 두지 않는다 (단, sub-bullet 또는 단락이 끼어드는
  경우만 빈 줄).
- sub-bullet 은 정확히 2칸 들여쓰기 (`  -`).
- lazy continuation (들여쓰기 없이 다음 줄에 본문 이어쓰기) 은 사용하지 않는다.

### 7. 강조 (emphasis)

- **굵은 글씨** (`**bold**`) — 라벨 또는 핵심 결론 한두 단어에만 사용. 한 단락
  안에 굵은 글씨가 3개 이상 등장하면 강조의 의미가 사라진다 — 그런 경우 강조를
  줄이거나 list 로 구조를 바꾼다.
- `backtick code` — 코드·명령·파일명·기술 용어.
- italic (`*text*`, `_text_`) 은 신중하게. 특히 Claude iOS 앱에서 italic 이 작게
  보이는 문제가 있으니 iOS 사용자에게 답변을 보낼 때는 굵은 글씨나 backtick 으로
  대체하는 게 안전하다.

### 8. 영문 용어 병기

전문 용어는 `한국어 (영문, 짧은 풀이)` 형태로 처음 등장할 때만 병기한다. 두 번째
등장부터는 한국어만 사용한다.
